# 本地双窗口测试工作流

## 结论
- 日常开发回归应优先使用本地双窗口。
- 真实双机测试保留给最终验收和网络/部署差异验证。

## 2026-04-11 最新状态
- `launch_opengl - local host.bat` 现已改为 `--fastmp=host --parallelturnpvphost`，不再只是打开多人菜单，而会自动走到 PvP host 调试建房流程。
- `launch_opengl - local both.bat` 现已改为等待 host 日志中出现 `DirectHost started on port 33771` 后再拉起 join，避免“客机先连、主机还没建房”的竞态。
- 但本地 fastmp 双窗口链路目前仍不能替代真实双机验收：动态测试已确认 host 能建房，join 失败点改为 ENet 握手 `net ID` 不匹配。
- 具体表现为：本地 `--fastmp=join -clientId=1001` 发送的是假 net ID `1001`，而当前 DirectConnectIP host 回给它的是主机 SteamID，最终报 `Could not connect / DisconnectionReason InternalError`。
- 因此本地双窗口目前适合做“模组是否启动、host 是否成功建房、UI/入口是否挂上”的快回归，不适合再当作 lobby/join 成功性的最终依据。
- 下一轮应切回真实双机 `DirectConnectIP` 测试，沿 `DirectClientConnectionInitializer netId=<SteamId> host=<ip>:33771` 这条真实链路验收。

## 为什么这条路更适合当前项目
1. 不需要副机部署和拉日志才能完成大多数回归
2. 仍然走游戏内多人链路，能覆盖当前 PvP mod 的大部分同步/回合问题
3. 迭代速度比双机高很多
4. 你当前已经有真实双机工作流，因此两条线可以并存：
   - 本地双窗口：开发主流
   - 双机：验收流

1. 运行 `launch_opengl - local host.bat`

### 方式二：一键双开
1. 运行 `launch_opengl - local both.bat`

## 当前观察
- 你游戏根目录里已有 `launch_opengl - client.bat`
- 该文件当前内容同时包含 `--fastmp=join` 和 `--fastmp=host`
- 这不是一个可靠的本地 join 启动参数组合
- 因此当前工作流不复用它，而是生成新的明确脚本

## 日志规则
1. 本地双窗口默认不要共用 `AppData\\Roaming\\SlayTheSpire2\\logs\\godot.log`
2. 当前启动器已为每个窗口指定独立日志：
   - Host: `local_fastmp_logs\\host.log`
   - Join1001: `local_fastmp_logs\\join_1001.log`
   - Join1002: `local_fastmp_logs\\join_1002.log`
3. 因此不会出现“只剩一端日志”或“两端日志混到同一个文件里”的问题
4. 需要注意：日志分离了，但**存档/用户数据仍然是同一个 Windows 用户目录**

## 推荐策略
1. 日常开发：本地双窗口用于启动/入口/host建房快回归
2. 联机成功性与完整 PvP 流程：真实双机 `DirectConnectIP` 复测
3. 双机桌面 MCP：保留为后续“完全自动化”路线，不再阻塞当前开发
