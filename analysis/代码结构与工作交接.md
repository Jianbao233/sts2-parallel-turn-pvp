- `src/ParallelTurnPvp/Bootstrap`

- `src/ParallelTurnPvp/Core`

  - 架构、设计、里程碑、变更、交接文档
  - 测试部署工作流见 [测试部署工作流.md](K:\杀戮尖塔mod制作\STS2_mod\PVP_ParallelTurn\analysis\测试部署工作流.md)

## 当前关键文件
- `Bootstrap/ParallelTurnPvpArenaBootstrap.cs`
  - 进入调试模式、配置固定测试局
- `Core/PvpModels.cs`
  - `PvpMatchRuntime`、`PvpRoundState`、有限意图 runtime 的主宿主
- `Core/PvpNetBridge.cs`
  - 当前 authoritative 消息桥；注意：当前只更新 runtime，不再写回原版 live combat
- `Core/PvpExecutionPlanner.cs`
  - 将 `PlanningSubmission` 编译为阶段化 `RoundExecutionPlan`，是后续延迟结算的第一层执行桥
- `Core/PvpDeltaPlanner.cs`
  - 将 `RoundExecutionPlan` 编译为结构化 `RoundDeltaPlan`；当前白名单动作语义已经从“卡名判断”拆成统一 delta 操作
- `Core/PvpDelayedExecution.cs`
  - 第一批延迟应用实验桥。代码骨架保留，但 live apply 已因 replay/checksum 边界问题被 feature flag 关闭，当前不会再在切边时回写 vanilla combat
- `Core/PvpPredictionEngine.cs`
  - 基于 `RoundDeltaPlan` 预测白名单动作的回合末快照；当前只做预测与对比，不回写 vanilla live combat
- `Core/PvpRoundResolver.cs`
  - 现在已经改为 `PlanningSubmission + RoundExecutionPlan + RoundDeltaPlan + PredictedFinalSnapshot` 驱动的回合摘要桥，后续可继续演进成真正延迟结算器
- `Core/ParallelTurnFrontlineHelper.cs`
  - 前线优先目标、前线/本体关系、frontline 查询
- `Patches/CombatPatches.cs`
  - 战斗主补丁：目标规则、回合切换、overlay 挂载、视觉层整理、胜负流程
- `Patches/RoundTrackingPatches.cs`
  - 动作记录、锁回合、结束回合撤销拦截
- `Ui/ParallelTurnIntentOverlay.cs`
  - 首版有限意图 overlay
- `tools/PullSecondaryLogs.ps1`
  - 从 `\\DESKTOP-U51KJJ2\SlayTheSpire2\logs\godot.log` 拉回副机日志
- `tools/desktop_mcp/mcpcontrol/Setup-McpControlSecondary.ps1`
  - 副机桌面控制接入的一次性安装脚本；会安装 Node/Python/VS BuildTools 与 `mcp-control`

## 当前已知技术债
1. 首锁奖励只保留设计，未恢复 live 实现
2. `PvpRoundResolver` 仍然只消费计划并生成摘要，没有真正驱动完整伤害/治疗/死亡结算
3. `FrontlineBrace / FrontlineSalve` 的 live delayed-apply 试验已关闭；目前仍回到即时执行，后续迁移不能再通过切边时直接改 vanilla live combat 来实现
4. `PvpDeltaPlanner/PvpPredictionEngine` 目前只覆盖白名单调试内容和当前前线规则，仍不是最终规则执行器
5. 有限意图 UI 仍是 debug overlay，不是最终交互层
6. 当前系统仍对 `BattlewornDummy` 壳有依赖
7. 副机部署采用共享目录直连，不再依赖手工复制与手工转存日志
8. 双机桌面控制的最终路线固定为 `MCPControl SSE`；`Windows-MCP.Net` 不用于副机直连
